# 互換契約と設定分離仕様

## 目的
AI による変更で事故が起きやすいのは、公開面の破壊と設定の混線である。
この文書では、その境界を定義する。

---

## 1. 互換契約とは何か
互換契約とは、「内部実装をどう変えてもよいが、これだけは壊してはいけない」という公開面の定義である。

---

## 2. 契約対象

### API
- path
- method
- required keys
- optional keys
- nullability
- enum 値
- status code

### DOM
- `data-controller`
- `data-action`
- `data-*`
- CSS class を外部 JS が前提にしている場合はそれも含む

### CLI
- コマンド名
- 引数
- exit code
- stdout / stderr の意味
- `--json` がある場合はその shape

### DB
- 既存カラム名
- 型
- null 制約
- index
- 外部連携が依存する値

### Job / Webhook
- payload shape
- retry 前提
- event 名
- queue 名

---

## 3. 契約文書の置き場

### 推奨
- `docs/contracts/api.md`
- `docs/contracts/dom.md`
- `docs/contracts/cli.md`
- `docs/contracts/jobs.md`

### 原則
- 重要な公開面はテストにも反映する
- 文書だけに依存しない
- テストだけにも依存しない

---

## 4. 変更分類

### additive change
既存利用者を壊さずに追加する変更。
例:
- optional key の追加
- 新コマンド追加
- 既存挙動を維持した内部整理

### breaking change
既存利用者を壊す変更。
例:
- key rename
- nullability 変更
- 既存コマンド削除
- exit code 意味変更
- DOM attribute rename

### ルール
- additive は自動で進めうる
- breaking は停止して人間レビュー

---

## 5. 設定分離

## 共有設定
repo に置く。
例:
- `.env.example`
- `config/*.yml`
- `docs/` 内の仕様
- `bin/*`
- CI workflow

## 個人設定
repo に置かない。
例:
- `.env.local`
- 個人 token
- IDE 設定
- 個人用 MCP 接続先
- 個人の実験設定

## 本番 secrets
repo に置かない。
例:
- API key
- 秘密鍵
- DB password
- session secret
- third-party credentials

---

## 6. `.env.example` の原則
- 必要最小限のみ書く
- ダミー値を入れる
- 値の意味をコメントで示す
- 本番値を入れない
- ローカルで必要なものだけに絞る

---

## 7. Codex に対する指示原則
設定や契約を触らせるときは、以下を明示する。

- 何が public contract か
- 何が internal detail か
- どこまで repo に含めるか
- どこからが秘密情報か
- 破壊的変更時は停止すること

---

## 8. 典型的な事故
- `.env.example` に本番 URL を入れる
- `.env.local` 前提の値をコードに埋め込む
- data attribute rename を内部変更だと思って進める
- CLI 出力変更で他スクリプトを壊す
- migration を軽微変更だと思って自動で進める

---

## 9. 完了条件
契約・設定タスクの完了条件は以下。

- [ ] 公開面が明文化されている
- [ ] 必要ならテストがある
- [ ] secrets が repo に入っていない
- [ ] `.env.example` が更新されている
- [ ] additive / breaking の分類が説明されている
