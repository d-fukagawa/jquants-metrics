## 仕組み 1: 検証ループ (Phase 1 + 2)

**問題**: 人間と Codex が違う方法で「動くか」を判定すると、緑なのに本番で壊れる。

**仕組み**:
- 入口を 1 本に統一: `bin/verify`
- 中身は責務別に分割: `bin/lint` / `bin/test` (これも `bin/verify` から呼ばれる)
- `npm run verify` / `make verify` / GitHub Actions の `bin/verify` がすべて同じパスを通る

**機能の流れ**:

```
bin/verify
  │
  ├─ bin/lint
  │    ├─ biome check .       ← コードスタイル / バグパターン
  │    └─ tsc --noEmit        ← 型整合 (Rails の zeitwerk:check 相当)
  │
  ├─ bin/test
  │    └─ vitest run          ← 271 tests / 29 files
  │
  └─ npm run build
       └─ vite build          ← Hono ルート登録 + Workers バンドル成立
                                (Rails の `bin/rails routes` 相当)
```

どれかが exit ≠ 0 で全体が止まる。CI 上でも同じ。緑 = PR 出せる状態。

**効く場面**:
- Codex が「テストは通った」と言うだけでなく、実際に biome / tsc / build まで通った状態を完了条件にできる
- `bin/verify` 緑 + push → CI も緑 が成立しているので、CI だけ落ちる事故は起きにくい

## 仕組み 2: 設定ファイルの責務分担

| ファイル | 役割 | 触れる頻度 |
|---|---|---|
| `biome.json` | lint ルール (formatter は無効) | 既存コード修正時のみ |
| `tsconfig.json` | TS 型検査の lib (DOM 含む) と jsx (hono/jsx) | ほぼ不変 |
| `.nvmrc` | Node バージョン固定 (`22`) | Node メジャー上げ時 |
| `wrangler.jsonc` | Cloudflare Pages ビルド設定 / Cron | infra 変更時 |
| `worker-configuration.d.ts` | `npm run cf-typegen` で再生成 (gitignore) | 自動 |
| `.dev.vars.example` | ローカル開発に必要な env のひな型 | 必須 binding 追加時 |
| `.dev.vars` | 個人用 secret (gitignore) | 各自の環境でのみ |
| `package.json` scripts | `bin/*` への薄いラッパー | bin/* と整合させる時 |
| `Makefile` | `bin/*` への薄いラッパー (Unix 派向け) | bin/* と整合させる時 |

**狙い**: 真実は `bin/*` と `*.json` の 1 箇所だけ。複数ファイルに同じ意図が書かれてズレる事故を避ける。

## 仕組み 3: GitHub Actions の二系統

```
.github/workflows/
├── verify.yml           ← (今回追加) PR/push トリガー、secret 不要
│
├── daily-sync.yml       ← Cron / scripts/daily-prices-sync.ts
├── financial-sync.yml   ← Cron / scripts/backfill-financials.ts
├── edinet-watch-sync.yml← Cron / scripts/edinet-watch-sync.ts
├── backfill-prices.yml  ← Cron / scripts/backfill-prices.ts
└── backfill-financials.yml ← 月次 backfill
```

| 系統 | 目的 | 失敗時 |
|---|---|---|
| **verify.yml** | コード品質ゲート | PR がブロック |
| **sync 系 5 本** | 本番データ同期 | Discord 通知 (`if: always()`) |

これらは互いに独立。verify が壊れても sync は走る、sync が壊れても verify は走る。両方とも `actions/checkout@v5` + `setup-node@v5` で統一。

## 仕組み 4: 互換契約 (Phase 3)

**問題**: Codex は内部リファクタのつもりで、外部から依存されている面を壊しがち。

**仕組み**: 公開面を 5 文書で明文化し、breaking / additive を区分。

```
doc/contracts/
├── api.md   ← POST /api/sync の shape, ルート一覧
├── cli.md   ← scripts/* の env, exit code, stdout プレフィックス
├── db.md    ← テーブル別の外部接点カラム
├── jobs.md  ← Cron / Actions のスケジュール, secret 名
└── env.md   ← Bindings vs Secrets vs process.env の境界
```

**読まれるタイミング**:
1. AGENTS.md / CLAUDE.md の First Read で参照される (Codex が初めにロード)
2. AGENTS.md の Non-Negotiable に「Do not break public surface defined in doc/contracts/」を追加済み
3. Codex が DB rename / API shape change / cron 時刻変更 を検出したら停止 → 人間レビュー

**効く例**:
- service 関数の rename → 内部だけなら additive (テスト合わせて OK)
- `scripts/daily-prices-sync.ts` の rename → workflow が参照しているので **breaking → 停止**
- `stock_master.code` カラム rename → sync が書き込み先にしているので **breaking → 停止**

## 仕組み 5: 変更が起きるときの典型パターン

### A. UI / ロジック変更 (例: 新しいスクリーニング条件追加)

```
1. src/services/screenService.ts に追加
2. src/routes/screen.tsx を更新
3. テスト追加
4. bin/verify  ← ローカルで緑
5. commit & push
6. verify.yml が CI で同じ bin/verify を実行 ← 緑
7. merge & Cloudflare Pages 自動デプロイ
```

→ 公開面 (URL, JSON shape) を変えていないので contracts 文書は更新不要。

### B. 同期スクリプトの env 追加 (例: 新しい retry 設定)

```
1. scripts/X.ts に新 env を default 付きで追加 (additive)
2. doc/contracts/cli.md に追記
3. .github/workflows/X.yml の workflow_dispatch.inputs に optional 追加
4. bin/verify
5. push
```

→ default があるので既存の Cron 実行は壊れない。additive change なので Codex も自動完了可。

### C. DB カラム追加 (例: financials に新指標)

```
1. src/db/schema.ts に NULL 可カラム追加
2. npm run db:generate  ← マイグレーション SQL 生成
3. **ここで停止して人間に渡す**
   理由: DB 変更は contracts/db.md の対象、Neon は共有インフラ
4. 人間が npm run db:migrate を実行
5. その後 service / route に反映 → bin/verify → push
```

→ harness 02 §8 の中断条件「migration の追加が必要」に該当。Codex は止まる。

### D. 公開面の breaking change (例: `/api/sync` の shape 変更)

```
1. Codex が POST /api/sync の response shape を変えようとする
2. doc/contracts/api.md と整合しない → 検出
3. **停止して人間に提案**
4. 人間が判断:
   a. 変更を許可するなら: contracts 更新 → 旧 shape の互換ラッパー追加 → 段階移行
   b. 却下するなら: 別アプローチ (新エンドポイント追加 = additive で代替)
```

## まとめ: 各仕組みの「効く粒度」

| 仕組み | 効く粒度 |
|---|---|
| `bin/verify` | コミット 1 つごと |
| verify.yml | PR / push ごと |
| `doc/harness/` | 設計判断のたび |
| `doc/contracts/` | 公開面に触れる変更のたび |
| `AGENTS.md` / `CLAUDE.md` | 各セッション開始時 (Codex が読む) |
| `.dev.vars.example` | 新規セットアップ / binding 追加時 |
| GitHub Secrets / Cloudflare Bindings | 本番環境 / sync ジョブ実行のたび |
| `worker-configuration.d.ts` (cf-typegen) | tsc 実行のたび (CI で自動再生成) |

人間も Codex も同じ「正解の状態」を判定できる、という構造になっている。
