# はじめに — 文書構成と使い方

## 目的
この文書群は、Codex などのコーディングエージェントに対して、Rails アプリ向けのハーネス設計・プラットフォームエンジニアリング実装を段階的に依頼するための分割ドキュメントである。

単一の長文で依頼すると、以下の問題が起きやすい。

- 目的と手段が混ざる
- 変更禁止領域が曖昧になる
- 互換契約や品質基準が抜ける
- CI / ローカル / 設定分離が一気に混線する
- Codex が「何をどこまでやれば完了か」を誤解する

そのため、責務ごとに文書を分ける。

## 文書一覧

### 1. `01_Codex依頼テンプレート.md`
Codex に直接渡す依頼文テンプレート。
- 目的
- 制約
- 完了条件
- 出力形式
- 中断条件

を固定する。

### 2. `02_ハーネス全体仕様.md`
ハーネス全体の設計方針。
- 評価ループ
- TDD 的検証
- lint / CI による外部基準化
- ローカル再現
- 設定分離
- 互換契約
- CLI ベースの安定した操作面

を定義する。

### 3. `03_CLIとローカル再現仕様.md`
`bin/*` や `Makefile` を中心とした操作面の仕様。
- `bin/setup`
- `bin/lint`
- `bin/test`
- `bin/verify`
- `bin/verify-changed`

などの責務と実装方針を定義する。

### 4. `04_CIと品質ゲート仕様.md`
GitHub Actions / 品質ゲート / リトライ方針。
- CI ジョブ
- fail 条件
- 差分検証
- 自動修正の限界
- 人間レビューへ返す条件

を定義する。

### 5. `05_互換契約と設定分離仕様.md`
公開面と設定境界の仕様。
- API / DOM / CLI / DB の互換契約
- `.env` / 秘密情報 / MCP 設定 / 個人設定の分離
- 破壊的変更の扱い

を定義する。

### 6. `06_段階導入計画.md`
一気に全部入れずに、どの順で導入するかを定義する。
- 最小導入
- 中間導入
- 本格導入

### 7. `07_実装マップ.md`
本プロジェクトでハーネスがどう実装されたか、各仕組みがどこで効くかをまとめた運用マップ。
- 全体像 (1 枚絵)
- 検証ループ / 設定ファイルの責務 / GitHub Actions 二系統 / 互換契約
- 変更が起きるときの典型パターン
- 「何かをやりたい」時の入口対応表

## 推奨の使い方

### 最初に読む順
1. `00_はじめに_文書構成と使い方.md`
2. `02_ハーネス全体仕様.md`
3. `03_CLIとローカル再現仕様.md`
4. `04_CIと品質ゲート仕様.md`
5. `05_互換契約と設定分離仕様.md`
6. `06_段階導入計画.md`
7. `07_実装マップ.md` (本プロジェクトの運用マップ)

### Codex に渡す順
Codex には常に以下の順で渡す。

1. `01_Codex依頼テンプレート.md`
2. 今回の作業対象の仕様文書
3. 必要なら `06_段階導入計画.md`

## 依頼単位の原則
1回の Codex 依頼で扱う責務は 1 つか 2 つまでに絞る。

### 良い依頼例
- `bin/verify` と `bin/lint` を実装する
- GitHub Actions に `bin/verify` を接続する
- `.env.example` と `docs/contracts/api.md` を追加する

### 悪い依頼例
- ハーネス全部を一気に実装する
- CI / lint / contract / secrets / deploy を同時に触る
- migration と認証と webhook をまとめて変更する

## この文書群の前提
- Rails アプリ
- CLI を共通の操作面にする
- 人間と Codex が同じ検証コマンドを使う
- 互換契約は口約束ではなく文書またはテストに落とす
- 設定と秘密情報を分離する
- 破壊的変更は自動で進めない

## 文書の更新ルール
- コマンドの真実は `bin/*`
- ルールの真実は各仕様文書
- README は概要のみ
- 実装が仕様から逸れたら、先に仕様を更新してから実装する
