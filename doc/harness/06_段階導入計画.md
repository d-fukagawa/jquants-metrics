# 段階導入計画

## 目的
ハーネスを一気に完成形で入れず、壊れにくい順で段階導入する。

---

## フェーズ1: 最小導入

### 目標
Codex に最低限の安全レールを与える。

### 導入対象
- `bin/setup`
- `bin/lint`
- `bin/test`
- `bin/verify`
- `Makefile`
- `01_Codex依頼テンプレート.md`

### 完了条件
- ローカルで `bin/verify` が通る
- Codex が「完了条件」を誤解しにくい
- 人間がレビューしやすい

### Codex に依頼する順
1. `bin/setup`
2. `bin/lint`
3. `bin/test`
4. `bin/verify`
5. `Makefile`

---

## フェーズ2: CI 接続

### 目標
ローカルで通る品質基準を CI に接続する。

### 導入対象
- `.github/workflows/ci.yml`
- DB 初期化
- `bin/verify` 連携

### 完了条件
- PR で CI が動く
- 失敗理由が追える
- CI とローカルの差が小さい

### Codex に依頼する順
1. CI workflow 追加
2. DB prepare 連携
3. JS / Node があれば install 連携

---

## フェーズ3: 契約と設定分離

### 目標
壊してはいけない面と、repo に置いてよい設定を明確化する。

### 導入対象
- `docs/contracts/`
- `.env.example`
- 設定境界の文書化

### 完了条件
- public contract が見える
- secrets が分離される
- 破壊的変更時の停止条件が明確

### Codex に依頼する順
1. `.env.example`
2. `docs/contracts/api.md`
3. `docs/contracts/cli.md`
4. `docs/contracts/dom.md` 必要なら

---

## フェーズ4: 高速化

### 目標
検証速度を改善し、開発中のイテレーションを速くする。

### 導入対象
- `bin/verify-changed`
- 差分 lint
- 差分 spec
- キャッシュ整理

### 完了条件
- 開発中の確認が速い
- 最終的には `bin/verify` に戻れる
- 高速化のために真実が分岐していない

---

## フェーズ5: 本格運用

### 目標
より広い変更を扱えるハーネスに育てる。

### 導入候補
- OpenAPI diff
- contract test
- CI job 分割
- 失敗要約の整備
- 変更ログテンプレート
- PR テンプレート

### 注意
この段階でも、認証・決済・migration・Webhook は厳格に扱う。

---

## 導入優先順位まとめ

| 優先 | 項目 | 理由 |
|---|---|---|
| 1 | CLI 統一 | 操作面が揃わないと他が全部ぶれる |
| 2 | ローカル再現 | 人間と Codex が同じ失敗を見られる必要がある |
| 3 | lint / test / verify | 完了条件を機械化できる |
| 4 | CI | 外部基準を常時適用できる |
| 5 | 設定分離 | 事故予防の効果が大きい |
| 6 | 互換契約 | 変更が広がるほど重要になる |
| 7 | 高速化 | まず正しさ、その後に速度 |

---

## やってはいけない進め方
- 最初から全部入れる
- Codex に巨大タスクを一発で渡す
- migration と認証と CI を同時に触らせる
- 仕様文書なしで「うまくやる」を期待する
- `bin/*` を整えずに workflow だけ先に作る
