# CIと品質ゲート仕様

## 目的
CI を「後から見るもの」ではなく、Codex の実装完了を判定する外部基準として使う。

---

## 1. 基本方針
- CI の入口は `bin/verify`
- ローカルと CI のコマンドを一致させる
- CI 専用分岐を増やしすぎない
- 品質ゲートは明示的に fail させる

---

## 2. 最小ジョブ構成

### verify job
必須ジョブは 1 つから始める。

#### 処理
1. checkout
2. Ruby セットアップ
3. Node セットアップ
4. JS install
5. DB prepare
6. `bin/verify`

### 理由
- 入口が 1 つだと Codex も人間も迷いにくい
- ジョブ分割は後から可能
- 初期段階では運用の単純さを優先する

---

## 3. 品質ゲート

### 必須ゲート
- rubocop
- brakeman
- rspec
- zeitwerk:check
- routes 確認

### 任意ゲート
- eslint
- typecheck
- system test
- OpenAPI diff
- contract test

### 判定
1 つでも失敗したら fail とする。

---

## 4. リトライ方針

### Codex 側のルール
- 同一失敗で 3 回連続失敗したら停止
- ログ全文ではなく要点を返す
- flaky の疑いがある場合は「再現性不明」と明示する

### 人間側のルール
- flaky を放置しない
- 「たまたま通った」は完了にしない
- CI だけ失敗する場合はローカル再現手順の不足を疑う

---

## 5. 差分検証

### 目的
作業中の試行速度を上げる。

### 仕様
- 開発中は `bin/verify-changed`
- PR 前は `bin/verify`
- CI では原則 `bin/verify`

### 禁止
- 差分検証だけで完了扱い
- changed files 判定が曖昧なまま本番運用
- 互換契約チェックを差分だけで済ませる

---

## 6. 自動修正の限界

以下は Codex が自動で完了扱いにしてはいけない。

- migration の追加・変更
- 既存 API の互換性に影響
- CLI 出力仕様の変更
- セキュリティ設定の大規模変更
- deploy workflow 変更
- 認証 / 認可の変更

これらは CI が green でも、人間レビューが必要。

---

## 7. GitHub Actions の設計原則
- workflow は薄くする
- 実行ロジックは `bin/*` に置く
- secrets は workflow に直書きしない
- 依存関係のキャッシュは使うが、キャッシュ前提でないこと
- DB 初期化は workflow 内で明示する

---

## 8. 完了条件
CI 整備タスクの完了条件は以下。

- [ ] `pull_request` で動く
- [ ] `bin/verify` が実行される
- [ ] ローカルと大きく乖離しない
- [ ] DB などの前提条件が明示されている
- [ ] 失敗時に原因が追いやすい

---

## 9. 将来拡張
段階導入後に以下を追加できる。

- job 分割
- matrix build
- security scan の細分化
- contract diff
- preview 環境チェック
- changed files に応じた job スキップ

ただし、最初から入れすぎない。
