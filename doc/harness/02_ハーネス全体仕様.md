# ハーネス全体仕様

## 目的
このハーネスは、コーディングエージェントの出力品質を、人間の勘やその場の対話だけに依存させず、再現可能な検証ループに乗せることを目的とする。

---

## 1. 評価ループ

### 要件
実装は以下のループに従う。

1. 目的を定義する
2. 実装する
3. 検証コマンドを実行する
4. 失敗を要約する
5. 修正する
6. 上限回数に達したら停止する

### 仕様
- 検証入口は `bin/verify`
- 差分を速く回す入口は `bin/verify-changed`
- 同一失敗で 3 回連続失敗したら停止
- 停止時は「何が詰まったか」を定型フォーマットで報告

### 禁止
- 失敗ログ全文をそのまま次の入力に流し続ける
- 無限リトライ
- 失敗理由のないテスト書き換え

---

## 2. TDD 的検証

### 要件
仕様変更または不具合修正では、実装前または実装と同時にテストを追加・更新する。

### 仕様
- 単純な内部リファクタで挙動が不変なら既存テスト通過で可
- 挙動変更がある場合は request spec / model spec / service spec を追加
- snapshot のみで完了にしない
- 「実装に合わせてテストを雑に変える」行為は禁止

### 原則
- unit: ロジック
- request: API / controller の外形
- system: 人間の重要導線のみ

---

## 3. lint / CI による外部基準化

### 要件
品質基準は、曖昧なレビューコメントではなく、実行可能な外部基準に寄せる。

### 仕様
- Ruby: `rubocop`
- Security: `brakeman`
- Test: `rspec`
- Frontend がある場合: `eslint`, `typecheck`
- CI は `bin/verify` を呼ぶ

### ルール
- lint disable は例外
- 例外はコメントで理由を書く
- CI green を完了条件に含める

---

## 4. ローカル再現

### 要件
人間と Codex が同じ方法で失敗を再現できること。

### 仕様
- 初期化は `bin/setup`
- 最終検証は `bin/verify`
- 軽量確認は `bin/verify-changed`
- 必須バージョンは `.tool-versions` または同等物に固定
- DB 準備は `bin/rails db:prepare` で再現できるようにする

### 禁止
- README にしか書かれていない手順
- CI でしか動かない専用処理
- 個人 PC 依存の分岐だらけのセットアップ

---

## 5. 設定分離

### 要件
共有設定・個人設定・秘密情報を分離する。

### 仕様
- repo に置く: `.env.example`, 共有設定ファイル, 仕様文書
- repo に置かない: 個人 token, 本番 secrets, 個人用 MCP 設定
- 環境変数は命名規則を揃える
- 共有設定の真実は repo に置く

---

## 6. 互換契約

### 要件
壊してはいけない公開面を先に定義する。

### 対象
- API
- DOM / data attributes
- CLI 出力
- DB schema
- background job 入出力
- webhook payload

### 仕様
- 互換契約は `docs/contracts/` に置く
- 重要な公開面はテストでも守る
- 破壊的変更は自動で進めない
- additive change と breaking change を分けて扱う

---

## 7. CLI ベースの安定した操作面

### 要件
人間と Codex が同じ操作面を使う。

### 仕様
- 正式な入口は `bin/*`
- `make` は薄い別名とする
- 各コマンドは exit code を正しく返す
- 1 コマンド 1 責務を守る
- `bin/verify` は最終検証だけを担当する

### コマンド一覧
- `bin/setup`
- `bin/lint`
- `bin/test`
- `bin/verify`
- `bin/verify-changed`

---

## 8. 中断条件

以下に該当する場合、Codex は停止し、人間に返す。

- migration の追加や変更が必要
- 認証・認可・決済・Webhook に触れる
- 既存 API / DOM / CLI に破壊的変更が出る
- secrets が必要
- スコープ外の大規模整理が必要
- CI の失敗原因が環境差異で切り分け不能

---

## 9. 成果物の定義
各タスクの成果物は以下の 4 つで構成する。

1. 実装
2. テスト
3. コマンド整備
4. 必要最小限の文書更新

これにより、単なる差分ではなく、再実行可能な変更にする。
